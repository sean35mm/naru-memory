Naru Memory — Implementation Plan

A local-first, harness-agnostic memory system for AI agents and developer workflows. Durable long-term memory, automatic fact extraction, hybrid retrieval, temporal graph reasoning, typed APIs, a CLI, a local tRPC server, and an OpenCode adapter — without any hosted dependency by default.

Naru Memory is a portable temporal fact graph with vector, text, entity, and graph retrieval indexes. Canonical memory records are local, inspectable, rebuildable, and separate from optimized indexes.
Local-first Harness-agnostic SQLite / libSQL TypeScript · tRPC
System architecture overview

1Objective

Build Naru Memory as a local-first, harness-agnostic memory system for AI agents and developer workflows.

Naru Memory should provide durable long-term memory, automatic fact extraction, hybrid retrieval, temporal graph reasoning, typed APIs, a CLI, a local tRPC server, and an OpenCode adapter. It should work without OpenCode, without MCP, and without any hosted dependency by default.

Naru Memory is a portable temporal fact graph with vector, text, entity, and graph retrieval indexes. Canonical memory records are local, inspectable, rebuildable, and separate from optimized indexes.

4Research Summary

4.1 HammerTime Findings

HammerTime's topology system was the strongest architecture reference. Key lessons:

Canonical data is portable

Source of truth should be rebuildable and exportable.

Indexes are derived state

Query engines and indexes are not the source of truth.

Manifests = idempotent commits

Boundaries for safe ingestion.

Raw vs synthesized overlays

Keep extraction and overlays separate.

Direct edges for fast traversal

Precomputed edges speed up graph queries.

Reified metadata for provenance

Evidence, scoring, and provenance are first-class.

Typed saved queries > raw query langs

Product APIs over SPARQL/Cypher/SQL.

Streaming + batching at scale

Graph growth needs streaming handling.

Sidecars create lifecycle complexity

Locks, pid files, health checks, stale cleanup.

→

Implication for Naru: store canonical memory in local tables and portable artifacts; treat FTS/vector/entity/current-view indexes as rebuildable; expose product APIs like context.build, memory.search, fact.neighborhood; avoid a required sidecar for the default embedded mode.

4.2 Graph DB Research

4.3 Mem0 Findings

Mem0 is the closest serious product reference. It solves broadly the same category: long-term AI agent memory.

What Mem0 does well — borrow

  • Single-pass ADD-only extraction
  • Extracts memories instead of raw-saving conversations
  • Treats assistant recommendations and actions as first-class memories
  • Multi-signal retrieval: semantic, BM25, entity, optional reranking
  • Entity linking creates graph-like retrieval boosting
  • Strong prompt rules: preserve proper nouns, dates, numbers, evidence
  • Relative temporal references grounded against observation date
  • Scope model: user, agent, app/project, run/session
  • CLI with agent-friendly JSON mode
  • OpenCode plugin uses native tools/hooks instead of MCP
  • Compaction handling preserves memory across summarization
  • Secret redaction before capture/injection

Where Naru should diverge

  • Mem0 OSS stores memories in vector store payloads; Naru stores canonical memory in SQLite/libSQL tables
  • Mem0 graph memory is mostly schema-free entity-to-memory linking; Naru models typed facts, evidence, temporal validity, and supersession
  • Mem0's platform/cloud path is central; Naru is local-first by default
  • Mem0 update/delete semantics are more direct; Naru uses supersession plus explicit privacy deletion
→

Implication for Naru: Borrow Mem0's extraction and retrieval quality. Borrow the adapter UX patterns. Do not copy the canonical data architecture.

9Scope Model

Use a typed scope graph rather than a single string.

9.1 Scope Types

ScopePurpose
userDurable personal preferences and operator habits across projects
workspaceLocal machine/workspace-level facts
projectRepo/project conventions, decisions, architecture, domain facts
branchBranch-specific task state and implementation notes
sessionCurrent agent run or conversation
agentAgent-specific behavior, tools, constraints, or learned operations
globalExplicit all-project alias for the current user; resolved as a query-time scope-set expansion, not a stored scope node, and not a normal write target
→

global is not a row in scopes. It is a query-time directive that expands the allowed read set to every project scope belonging to the current user. Writes never target global; a "global" write resolves to the user scope.

9.2 Default Write Scope

9.3 Default Read Order & Ranking

For normal developer work, context.build reads in this order, with closer scopes weighted higher. agent scope is read alongside session/branch — it is run-adjacent but persists across sessions, so it sits just below session:

Read order (left → right) — closer scopes are read first and weighted higher
!

Global reads should require explicit user or tool intent (e.g., "Search all projects" or --scope global). global only enters ranking when a query opts in; its members rank by their underlying project/user scope. Global writes resolve to user and should be rare and explicit.

9.4 Scope Safety Rule

Scope filtering must happen before ranking and before graph expansion.

✕ Unsafe pattern

rank all memories -> filter scope

Memory from outside the allowed scope may influence ranking before it's removed.

✓ Safe pattern

resolve allowed scope set
  -> candidate retrieval inside allowed scopes
  -> graph expansion inside allowed scopes
  -> rank
Safe retrieval pipeline (scope-aware at every hop)

10Raw Episode Text Retention

Default recommendation: redacted episode retention.

10.1 Retention Modes

ModeBehaviorDefault
redactedStore redacted episode text, evidence snippets, and source hashesyes
minimalStore facts and evidence hashes only, no episode bodyno
encryptedStore original source encrypted locallylater
noneStore extracted facts only, no episode/evidence textadvanced
!

Retention determines what is rebuildable. The core principle that derived indexes are droppable and regenerable from canonical tables (§12.2) only fully holds under redacted/encrypted. Under minimal/none the source text is gone, so facts cannot be re-extracted and embeddings cannot be regenerated — only FTS/entity indexes over already-stored facts can be rebuilt. Choosing minimal/none is an explicit trade of rebuildability/auditability for privacy, surfaced in naru status and at config time.

10.2 Why Redacted Default

Redacted retention is the best default because it balances privacy and auditability.

Inspect why a fact exists

Users can audit extracted memories.

Re-extraction

Re-run extraction when prompts or schema improve.

Debug false memories

Trace hallucinations back to source.

Preserve provenance

Without storing secrets.

No chat-log archive

Avoid turning Naru into a transcript dump.

Source hashes

For dedupe and provenance.

Redaction boundary — secrets never cross into storage, indexes, or LLM/embeddings

11SQLite Schema Plan

The exact SQL should be finalized during implementation, but the initial schema should cover the following tables.

Schema table map — canonical (solid) vs derived (dashed)

11.1 schema_migrations

version · name · checksum · applied_at

11.2 scopes

id · type · name · key (unique) · parent_scope_id · metadata_json · timestamps. Indexes: unique type+key, parent_scope_id.

11.3 episodes

id · scope_id · source_type · source_ref · source_hash · hmac_hash · retention_mode · redacted_text · metadata_json · observed_at · created_at. Indexes: scope_id, observed_at; source_hash; optional unique scope_id + source_hash.

11.4 entities

id · scope_id · type · canonical_name · normalized_key · aliases_json · metadata_json · timestamps. Indexes: scope_id, normalized_key; type, normalized_key. Per-scope by default: scope_id is populated for normal entities; a null scope_id denotes an explicitly promoted global/shared entity (the exception, never the default). Shared entities are the primary leakage vector — see §18.3.

11.5 facts

id · scope_id · subject_entity_id · predicate · object_entity_id · object_value · statement · statement_hash · confidence · status · valid_from · valid_to · observed_at · timestamps · metadata_json. Status: active · superseded · deleted · rejected · archived.

11.6 evidence

id · fact_id · episode_id · span_start · span_end · redacted_quote · quote_hash · extractor_name · extractor_version · created_at. Indexes: fact_id, episode_id, quote_hash.

11.7 edges

id · scope_id · source_type · source_id · predicate · target_type · target_id · confidence · metadata_json · created_at. Indexes: scope_id, source_type, source_id; scope_id, target_type, target_id; scope_id, predicate.

11.8 supersessions

id · old_fact_id · new_fact_id · reason · confidence · created_at. Indexes: old_fact_id, new_fact_id, unique old_fact_id + new_fact_id.

11.9 embeddings

id · target_type · target_id · provider · model · dimension · source_hash · status · vector_ref · timestamps · error. Vector storage may live in side tables depending on driver.

11.10 index_state

index_name (PK) · index_version · source_watermark · source_hash · status · last_rebuilt_at · error · metadata_json.

11.11 FTS Tables (derived)

FTS5 for BM25-style retrieval. Candidates: facts_fts(fact_id, statement, predicate, entity_text, scope_key) and entities_fts(entity_id, canonical_name, aliases). These are derived and rebuildable.

→

Identifier & hash strategy. Dedup-on-import and cross-machine portability depend on stable IDs and hashes. IDs: ULID (or UUIDv7) for all primary keys — sortable, collision-resistant, offline-generable, stable across export/import. statement_hash: a portable content hash over a canonicalized statement, computed identically everywhere — Unicode NFC, trim + collapse whitespace, casefold, deterministic serialization of (scope_key, subject, predicate, object/value) with entity refs resolved to normalized keys (not raw IDs). Bump a hash_version when the rule changes. source_hash follows the same discipline over redacted source + metadata.

12Storage and Indexing Strategy

12.1 Canonical Store

The canonical store includes: scopes, episodes, entities, facts, edges, evidence, supersessions. These tables are authoritative and should be backed up/exported.

12.2 Derived Indexes

FTS5 / BM25

Full-text ranking.

Entity alias lookup

Fast entity resolution.

Current-view active fact

Filter out superseded/deleted.

Graph-neighborhood

Helper indexes for traversal.

Vector index

When supported.

Context cache

Optional, later.

↻

Derived indexes must be rebuildable from canonical tables, with two caveats: facts are only rebuildable from source under redacted/encrypted retention (§10.1); vector indexes rebuild only when the source/fact text is retained and the embedding provider/model is available, at real time/compute cost — treat vector rebuilds as a deliberate operation, not a routine drop-and-recreate.

12.3 SQLite Operational Settings

WAL mode

Write-ahead logging; concurrent readers throughout.

Busy timeout

Avoid spurious lock failures.

Foreign keys enabled

Enforce referential integrity.

Transactional writes

Atomic ingestion.

Restrictive file perms

Local memory DB protection.

!

Write coordination (not optional). Three write entry points — CLI embedded, the tRPC server, and the OpenCode adapter — make concurrent writers a near-certainty, not an edge case. Single logical writer: when a local server is running, all writers proxy writes to it (it owns a serialized ingestion queue), discovered via a server.json file (host, port, token, pid) at a known config path. Embedded fallback: with no live server, the CLI/adapter writes in embedded mode guarded by a file lock. WAL keeps readers concurrent throughout; this rule and the auth token in §15.3 are the same mechanism viewed from storage vs. transport.

13Ingestion Pipeline

The default ingestion pipeline should be ADD-only at the storage boundary.

Ingestion pipeline (ADD-only at storage boundary)

13.1 Episode Capture

Inputs: source text or structured messages; source type (chat · tool · summary · import · manual · document · system); scope; observation timestamp; metadata.

Rules: redact first; validate scope; store source hash; deduplicate identical episodes per scope where appropriate.

13.2 Extraction

!

Latency budget: reads sync, writes async. Extraction is an LLM call and must never sit on a latency-sensitive path. Memory reads (search, context.build, prompt injection) are synchronous and must stay fast. Memory writes (capture → extraction → linking → embedding → indexing) run asynchronously in the background off the agent's critical path: capture acknowledges quickly once the episode is durably stored and redacted; new facts become retrievable once indexed. Essential for the OpenCode adapter, where a per-message hook (§17.4) cannot block the agent loop on a local model.

Extraction should use a local LLM provider when configured. The extractor should output typed JSON:

json{
  "facts": [
    {
      "subject": "User",
      "predicate": "prefers",
      "object": "dark mode",
      "statement": "User prefers dark mode for developer tools.",
      "entities": ["User", "dark mode"],
      "confidence": 0.86,
      "valid_from": null,
      "valid_to": null,
      "evidence": {
        "quote": "I prefer dark mode",
        "span_start": 0,
        "span_end": 18
      },
      "linked_fact_ids": []
    }
  ]
}

Prompt principles (borrowed from Mem0)

  • Extract all memorable information, not raw conversation.
  • Preserve proper nouns, titles, exact dates, numbers, and specific details.
  • Ground relative dates against observation date.
  • Extract content from shared documents, not the meta-action of sharing.
  • Distinguish user-stated facts from assistant-generated recommendations.
  • Skip greetings, filler, and generic acknowledgments.
  • Prefer rich, self-contained memories over overly atomic fragments.
  • Avoid fabrication.
  • Include evidence references where possible.

13.3 Local/Offline Fallback

ⓘ

When no extractor is configured: memory.add --infer=false stores supplied text as a fact-like manual memory; deterministic entity extraction may still run; FTS search remains available; system reports extractor: unavailable in status — it does not fail startup.

13.4 Entity Linking

  • Normalize names: lowercase, trim, collapse whitespace, strip punctuation where safe.
  • Match aliases within scope.
  • Match exact normalized key.
  • Optional embedding similarity for entity names when embeddings are configured.

Entity linking should be scope-aware.

13.5 Deduplication

Signals

  • Stable statement_hash within scope
  • Same subject/predicate/object
  • High similarity to existing active fact
  • Same evidence hash
  • Extractor-provided linked_fact_ids

Actions

  • Exact dup: attach evidence to existing fact or ignore per policy
  • Near dup w/ richer context: add new fact and optionally supersede old weaker fact
  • Conflict/change: add new fact and create supersession

Capability tiers — be explicit about what degrades. Distinguishing a near-duplicate from a changed/conflicting fact is AI-hard, so capability is tiered by what's configured (and the milestones reflect this):

TierAvailableDedup capabilityAuto-supersession
No LLM, no embeddings (M1)exact hash onlyexact statement_hash within scopenone — use manual naru supersede
Embeddings, no LLM (M3)+ semantic similarityexact + near-duplicate by vector similarityheuristic only; conflicts still need confirmation
LLM configured (M2+)+ semantic judgmentfullLLM-judged change/conflict detection
→

naru supersede / fact.supersede is the always-available manual fallback in every tier. Automatic supersession never silently fires below the LLM tier.

13.6 Supersession

Changed facts should not overwrite older facts.

Supersession timeline — non-destructive update

Result: insert new fact; mark old fact as superseded; insert supersessions(old_fact_id, new_fact_id, reason); current view returns the new fact; history view can still show both.

14Retrieval and Context Building

Retrieval should be hybrid from the start, even if vector retrieval is optional at first.

Hybrid retrieval & context building pipeline

14.1 Candidate Sources

SourcePurpose
FTS / BM25Exact terms, filenames, proper nouns, commands, errors
Entity matchPull memories connected to query entities
Vector KNNSemantic similarity
Graph neighborhoodRelated facts around matched entities/facts
Recency / current viewPrefer currently active facts when relevant

14.2 Ranking Signals

Scope priority
Semantic vector score
BM25 score
Entity match strength
Graph proximity
Fact confidence
Evidence quality
Temporal validity
Recency
Supersession / current status
User-pinned / protected (later)
→

Combination methodology. Signals are combined, not used in isolation. Normalize each to a comparable range first (BM25 and cosine are not the same scale). Combine as a weighted linear score with named, config-exposed weights as the v1 baseline. Apply scope priority and current-view/temporal validity as gates/multipliers, not just additive terms, so an out-of-scope or superseded fact can't be ranked in by a strong vector score. Emit per-signal contributions in the reason array (§14.4). Weights are tuned against the quality eval set (§21.7), not guessed — the defaults are a checked-in artifact produced by that eval.

14.3 Current View

Current view excludes: deleted facts, rejected facts, and superseded facts when the replacement is active and in an allowed scope. It may include superseded facts only when the query asks for history, asOf is before the supersession, or the API explicitly requests history.

14.4 Context Output

context.build should return structured context, not just a string:

json{
  "items": [
    {
      "fact_id": "fact_...",
      "statement": "User prefers dark mode for developer tools.",
      "scope": "user",
      "score": 0.91,
      "evidence_refs": ["ev_..."],
      "temporal": { "valid_from": null, "valid_to": null },
      "reason": ["entity", "bm25", "scope:user"]
    }
  ],
  "prompt_block": "...",
  "token_estimate": 412
}

Adapters can inject prompt_block, while tools and UIs can inspect items.

15.3 Server Requirements

⚠

Security is from the first release, not deferred. A localhost-bound server is reachable by any local process and, via DNS-rebinding/CSRF, by any web page the user visits — and it holds personal/project memory.

Local server should, from its first release:

  • Bind to loopback (127.0.0.1) only by default.
  • Require an auth token on every request. Generated at first start and written to server.json (mode 0600) at a known config path; clients read it from there.
  • Validate Origin/Host headers and reject cross-origin/unexpected-host requests (defends against DNS-rebinding and browser-driven access to 127.0.0.1).
  • Write server.json (host, port, token, pid) for discovery (§12.3), with restrictive permissions.
  • Avoid exposing non-loopback interfaces unless explicitly configured (and warn loudly if configured).
  • Support graceful shutdown and remove its server.json on exit (with stale-file detection on start).
  • Expose health and status (health may be unauthenticated; everything else requires the token).
  • Use the same core services as the CLI.

  • Uninstall should remove only owned entries.
  • Preserve user config.
  • Use restrictive permissions for local settings.
  • Do not enable MCP by default.

18Privacy and Security

18.1 Redaction

⚠

Redaction is best-effort defense-in-depth, not a hard guarantee. Pattern/regex redaction will miss secrets in custom or non-standard formats. Never treat "redacted" as proof no secret is present; downstream layers (embeddings, logs, prompt injection) must minimize blast radius — once a secret reaches a vector index it is effectively unrecoverable. Where feasible, prefer not capturing high-risk material at all over relying on redaction to scrub it.

Redact before: episode storage, fact extraction, embedding generation, FTS indexing, logs, and OpenCode prompt injection.

Initial secret redaction patterns

  • API keys
  • GitHub tokens
  • OpenAI-style keys
  • AWS access keys
  • Slack tokens
  • Bearer tokens
  • Private key blocks
  • .env style secret assignments
  • Password-looking fields
  • High-entropy token detection (catch-all for unknown formats) + configurable allowlist
  • Configurable custom denylist for org/project-specific secret shapes

Redaction rules

  • Redact before persistence
  • Redact before embeddings
  • Redact before logs
  • Redact before adapter prompt injection
  • Never embed raw unredacted episode text
  • Store source hashes for dedupe and provenance
→

PII vs. secrets. Naru deliberately stores personal facts ("User prefers dark mode") — that is the product's payload, kept local-only. That is distinct from secrets and sensitive PII, which must be scrubbed. Always redact: secrets/credentials (above) and sensitive PII with no memory value — emails, phone numbers, full street addresses, government/ID numbers, payment card numbers. Retain by design: non-sensitive preferences, conventions, and operator habits. The posture is local-first containment: personal facts never leave the machine by default, are inspectable, and are removable via memory.forget (§18.2). PII redaction runs in the same pre-persistence pass as secret redaction, with the same best-effort caveat.

18.2 Forget / Delete

memory.forget must support privacy deletion.

Selectors

  • fact ID
  • entity ID
  • episode ID
  • scope
  • text query with confirmation
  • before/after date

Must purge or invalidate

  • canonical facts
  • evidence
  • episodes if selected
  • edges
  • embeddings
  • FTS rows
  • entity links if orphaned
  • index/cache references
⚠

For privacy deletes, deletion can be destructive. Supersession is for normal memory evolution, not privacy deletion.

18.3 Scope Leakage Prevention

!

Potential issue: because entities can be promoted to global/shared (scope_id IS NULL, §11.4), a fact in project A and a fact in project B can both link to the same entity. Naive traversal from one fact → shared entity → the other fact leaks project B's memory into project A.

  • Graph traversal must check allowed scopes at every hop.
  • Traverse through a shared entity, but only collect facts whose scope_id is in the allowed scope set. A shared/global entity node being in-scope never authorizes returning foreign-scope facts attached to it.
  • Entity identity may be shared; entity-attached facts are always re-filtered by scope before entering candidate sets or ranking.
  • Promotion of an entity to global does not promote its facts; facts retain their original scope.

19Import and Export

Naru should support portable memory bundles.

Export includes

  • schema version
  • scopes
  • episodes according to retention mode
  • entities
  • facts
  • evidence
  • edges
  • supersessions
  • no derived indexes by default

Import behavior

  • validate schema version
  • preserve IDs when safe
  • dedupe by source/fact hashes (portable statement_hash/source_hash, §11.5)
  • rebuild indexes after import
!

Vector re-embed caveat. Rebuilding the vector index on import requires the embedding provider/model used at export (or a chosen replacement) to be available on the importing machine, and incurs re-embedding cost. If unavailable, import still succeeds and FTS/entity retrieval works; vector search stays degraded until re-embedding runs. Bundles should record the embedding provider/model/dimension so the importer can warn on mismatch.

→

This follows HammerTime's canonical-artifact principle.

21.7 Quality Evaluation (relevance, not just latency)

!

Benchmarks (§21.6) measure speed and size; they say nothing about whether retrieval returns the right memories or extraction produces correct facts — which is the actual product value. A small, version-controlled labeled eval set is required and run in CI as a quality gate.

Retrieval eval

Labeled (query, allowed-scope, expected fact IDs) cases. Metrics: recall@k / precision@k, scope-correctness rate (zero out-of-scope leaks is a hard pass/fail, §18.3), and ranking quality (MRR/nDCG). This is the ground truth for tuning the §14.2 ranking weights.

Extraction eval

Golden (episode → expected facts/entities) fixtures scored for precision/recall, plus proper-noun/date/number preservation checks (§13.2 prompt rules).

Supersession eval

Changed-fact fixtures verifying the right old fact is superseded and the current view returns the new one (no false supersessions).

CI gates

A build that regresses recall, leaks scope, or drops extraction precision fails. Track scores over time alongside the benchmark report.
